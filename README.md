## 该不该做

应该做，但首轮必须收敛。

RecallFlow 不应该先做成“大而全学习平台”，而应该先做成一个给家长和低年级孩子使用的 **题库 + 练习 + 背诵 + 复习追踪系统**。

核心场景是：

> 一年级孩子完成学校作业后，家长把当天需要巩固的内容录入或选择出来，孩子用 RecallFlow 做测试、背诵、拼读、口算和复习，系统记录掌握情况，并安排后续复习。

产品不只判断对错，还要用孩子能听懂的话解释“为什么”。例如 `12 + 24` 可以拆成 `10 + 20 = 30`、`2 + 4 = 6`，最后得到 `36`；组词题也要解释词语大概是什么意思、可以在什么句子里使用。

## 为什么

这个项目适合长期做，因为它同时满足三件事：

1. 真实需求明确：低年级孩子每天都有语文、数学、英语的重复练习需求。
2. 产品闭环短：录入内容、开始练习、提交答案、查看结果、安排复习。
3. 后续任务池大：题库、卡片、复习调度、错题本、统计、导入导出都能独立迭代。
4. 孩子需要即时解释：低年级孩子不能只看答案，需要看到拆步骤、讲意思、举例子。

需要避免的方向：

- ⚠️ 伪需求：一开始做 AI 出题、课程系统、社区、多人协作。
- ⚠️ 技术过度设计：一开始做复杂权限、同步、多端、重算法。
- ⚠️ 时间黑洞：题型无限扩展，但没有稳定学习流程。
- ⚠️ 收益不匹配：做很多炫酷页面，却不能帮助孩子完成练习和复习。
- ⚠️ 伪需求：趣味内容过多，反而干扰孩子理解题目。

## 一句话定位

RecallFlow 是一个面向低年级孩子的家庭学习复习系统，帮助家长管理题目和背诵内容，帮助孩子通过练习、解释、趣味反馈和间隔复习逐步掌握知识。

## 目标用户

### 主要用户

- 家长：录入学习内容、查看掌握情况、安排练习。
- 孩子：完成练习、背诵、测试和复习。

### 首轮不服务

- 老师班级管理
- 多学生机构场景
- 公开题库社区
- 复杂考试系统

- 首轮支持一个家长账号管理多个孩子。

## 登录与多孩子

首轮需要登录，但只做最小可行账号体系。

### 登录范围

- 手机号码 + 密码注册
- 手机号码 + 密码登录
- 退出登录
- 当前登录态保持
- JWT 由后端签发，前端通过 HttpOnly Cookie 保存
- 登录态默认 7 天过期
- 密码最少 8 位
- 手机号码必须唯一
- 首轮不做短信验证码
- 首轮不做找回密码

### 多孩子范围

- 新增孩子
- 编辑孩子
- 切换当前孩子
- 创建第一个孩子后自动设为当前孩子
- 每个孩子有独立练习记录、错题、每日目标和奖励记录
- 孩子只能在当前登录家长账号范围内选择
- 后端接口必须校验 `student_id` 属于当前登录家长

首轮不做老师、班级、组织等复杂权限。

## 后台

首轮需要最小后台，用于家长管理内容和查看数据。

后台不和孩子端共用同一个前端应用，建议独立为管理端应用。

后台范围：

- 孩子管理
- 题库管理
- 练习记录查看
- 错题查看
- 奖励记录查看
- 基础统计查看

首轮后台不是运营后台，不做平台级用户管理、审核、付费和复杂权限。

## 核心模块

### 1. 题库模块

管理原始学习内容。

字段可以包括：

- 题目标题
- 题干
- 标准答案
- 解析
- 儿童解读
- 趣味提示
- 学科
- 年级
- 知识点
- 标签
- 难度
- 来源
- 判分方式

可扩展方向：

- 搜索
- 筛选
- AI 一键导入
- 批量编辑
- 重复题检测
- 标签管理

### 2. 卡片模块

把题目转换成适合练习和复习的卡片。

常见卡片类型：

- 问答卡：题干到答案
- 背诵卡：提示到完整内容
- 拼音卡：汉字到拼音
- 组词卡：生字到词语
- 口算卡：算式到结果
- 翻译卡：中文到英文或英文到中文

卡片背面不只展示答案，还应展示：

- 简单解读
- 拆步骤说明
- 例句
- 趣味提示

可扩展方向：

- 一题多卡
- 卡片模板
- 卡片预览
- 卡片批量生成

### 3. 练习会话模块

负责孩子实际使用的主流程。

流程：

1. 选择今日练习
2. 展示第一张卡片
3. 孩子作答
4. 家长或系统判定结果
5. 展示答案、简单解读和趣味反馈
6. 记录本次表现
7. 自动进入下一题
8. 结束后展示结果

首轮可以用人工判定，避免过早做复杂自动批改。

判分策略：

- 数学口算：自动判分，首轮覆盖 1-2 年级单步基础加减乘除。
- 简单英语：文本匹配，首轮从字母、基础单词和简单口语表达开始。
- 语文组词：手动判定。
- 背诵内容：手动判定。
- 拼音题：首轮可以先手动判定，后续再做规则校验。

评分枚举：

- `wrong`：错误 / 不会。
- `unclear`：模糊。
- `correct`：正确。
- `mastered`：已掌握。

首轮数学自动判分只支持单步算式：

- `+`
- `-`
- `*`
- `/`

不做括号、多步混合运算和应用题自动判分。

练习模式：

- `daily_review`：今日练习。
- `homework_quiz`：作业后测试。
- `wrong_retry`：错题再练。
- `recitation`：背诵练习。

### 4. 复习调度模块

负责决定哪些卡片今天应该复习。

首轮规则可以很简单：

- 答错：当天再次出现，明天继续复习。
- 模糊：明天复习。
- 答对：3 天后复习。
- 连续答对：7 天后复习。

后续再扩展到更完整的间隔重复算法。

### 5. 错题本模块

低年级家庭场景里，错题本比复杂统计更重要。

应该支持：

- 查看错题
- 按学科筛选
- 按知识点筛选
- 重新练习错题
- 标记已掌握

### 6. 统计模块

首轮只保留家长真正会看的数据：

- 今日练习数量
- 今日学习时长
- 今日正确率
- 错题数量
- 连续练习天数
- 各学科掌握情况
- 每日目标题数完成情况
- 每日学习时长完成情况

后续再做趋势图、热力图和导出。

### 7. 每日目标模块

首轮支持两个简单目标：

- 每日目标题数
- 每日学习时长

默认值：

- 每日目标题数：`20`
- 每日学习时长：`15` 分钟

目标用于提醒、统计和触发轻量奖励。

### 8. 奖励激励模块

奖励是 MVP 必做，但首轮保持轻量。

可支持：

- 小奖牌
- 贴纸
- 小星星
- 积分

触发方式：

- 完成每日目标题数：小奖牌。
- 达到每日学习时长：贴纸。
- 连续答对 5 题：小星星。
- 错题再练后标记掌握：积分。
- 完成一次作业后测试：积分。

首轮只做展示和记录，不做兑换商城。

奖励不要做重。首轮只解决“孩子完成后有正反馈”，不做复杂成长体系。

## 解释与趣味反馈

这是 RecallFlow 区分普通题库的核心能力之一。

### 解释原则

- 面向一年级孩子，句子要短。
- 先讲步骤，再给结论。
- 多用具体例子，少用抽象术语。
- 数学题优先展示拆分过程。
- 语文题优先解释词语意思和使用场景。
- 英语题优先展示中文意思、读法提示和简单例句。
- 趣味描述要服务理解，不要喧宾夺主。

### 数学解释示例

题目：

```text
12 + 24 = ?
```

答案：

```text
36
```

儿童解读：

```text
先算十位：10 + 20 = 30。
再算个位：2 + 4 = 6。
最后把 30 和 6 合起来，就是 36。
```

趣味提示：

```text
可以把 12 想成 1 捆小棒加 2 根，把 24 想成 2 捆小棒加 4 根。
```

### 语文解释示例

题目：

```text
用“春”组词。
```

答案：

```text
春天、春风、春节
```

儿童解读：

```text
“春”常常和春季有关。春天是一个季节，春风是春天吹来的风，春节是中国很重要的节日。
```

趣味提示：

```text
想到花开、天气变暖，就很容易想到“春”。
```

### 英语解释示例

题目：

```text
apple 是什么意思？
```

答案：

```text
苹果
```

儿童解读：

```text
apple 是苹果。可以说：I eat an apple. 意思是：我吃一个苹果。
```

趣味提示：

```text
看到红红的苹果，可以想起 apple。
```

## 技术路线

### 当前选择：Next.js + FastAPI + PostgreSQL

- 用户端前端：Next.js + TypeScript
- 后台前端：独立 Next.js + TypeScript 管理端
- 后端：FastAPI
- 数据库：PostgreSQL
- ORM：SQLAlchemy 2.x
- 数据库迁移：Alembic
- 本地开发：Docker Compose

原因：

- Next.js 比纯 React 更适合做完整 Web 产品，页面路由、数据加载、构建部署都更完整。
- FastAPI 更适合后续接入 AI、人工导入解析、数据分析、统计任务和调度算法。
- 后续功能会持续增加，前后端分离更利于长期迭代。
- PostgreSQL 更适合承载题库、导入批次、练习记录、统计和奖励数据。
- 后端独立后，更容易扩展人工导入、AI 一键导入、统计任务和异步任务。
- Alembic 适合管理 PostgreSQL schema 迁移。
- Docker Compose 能降低 PostgreSQL 和本地开发环境的启动成本。

### 为什么不选纯 React

- 纯 React 需要额外选择路由、构建、数据加载方案。
- 项目后续页面会变多，Next.js 的项目结构更稳。
- 如果未来做服务端渲染、权限页、管理页，Next.js 更容易承接。

### 为什么不选 NestJS

- TypeScript 全栈一致。
- 工程结构规范，适合长期 API 项目。
- 但本项目后续很可能接入 AI、题目解析、导入清洗和统计分析。
- Python 生态在这些方向上更省事。

### 当前判断

前后端分离 + PostgreSQL 可以做，但首轮仍要控制功能边界。不要因为架构升级，把 MVP 做成平台化工程。

## 当前默认决策

### 功能范围

- 首轮支持登录。
- 登录方式为手机号码 + 密码。
- 密码最少 8 位。
- 手机号码必须唯一。
- 首轮不做短信验证码，放到 Phase 2 再确认。
- 首轮不做找回密码，放到 Phase 2 再确认。
- 认证使用后端签发 JWT，前端通过 HttpOnly Cookie 保存。
- 登录态默认 7 天过期。
- 首轮支持一个家长账号管理多个孩子。
- 创建第一个孩子后自动设为当前孩子。
- 孩子字段首轮只保留姓名和年级。
- 孩子只能在当前登录家长账号范围内选择。
- 后端接口必须校验 `student_id` 属于当前登录家长。
- 练习、目标、奖励、错题按孩子独立统计。
- 题目归家长，可选绑定孩子；卡片和练习记录归孩子。
- 不做老师、班级、组织等复杂权限。
- 首轮需要最小后台，用于孩子、题库、导入、练习记录、错题、奖励和基础统计管理。
- 后台是独立管理端应用，不和孩子端共用同一个前端。
- 采用混合判分：数学自动判分，简单英语文本匹配，语文和背诵手动判定。
- 首轮支持背诵内容。
- 首轮支持错题本。
- 儿童解读选填，不是每一题都必须填写。
- 趣味提示选填。
- 首轮支持每日目标，包含目标题数和学习时长。
- 每日目标默认值为 `20` 题和 `15` 分钟。
- 首轮支持奖励激励，包含小奖牌、贴纸、小星星、积分。
- 首轮以手动录入为主。
- JSON 导入放到 Phase 2，标准结构届时确认。
- AI 一键导入作为后续能力预留。
- 题目删除采用停用或归档，不做物理删除。
- 需要题目难度字段，首轮先使用简单三档：`easy`、`normal`、`hard`。

### 题型范围

- 语文先做：1-2 年级拼音、组词、生字、背诵。
- 数学先做：1-2 年级基础口算和基础加减乘除。
- 英语先做：最基础的字母、单词、中英互译、简单口语表达。
- 支持“作业后测试”独立模式。

题目类型枚举：

- `chinese_word`：语文生字。
- `pinyin`：拼音。
- `word_group`：组词。
- `recitation`：背诵。
- `math_arithmetic`：数学口算。
- `english_word`：英语单词。
- `english_oral`：英语口语。

### 技术栈

- 倾向前后端分离。
- 前端采用 Next.js + TypeScript。
- 后台采用独立 Next.js + TypeScript 管理端。
- 后端采用 FastAPI。
- 数据库采用 PostgreSQL。
- 使用 Alembic 管理数据库迁移。
- 使用 Docker Compose 管理本地开发依赖。
- PC 优先。
- 暂不支持离线可用。

### 产品形态

- 先做 Web 产品，后续可持续迭代更多能力。
- 家长手动录题为主。
- 首轮以练习工具为主，保留间隔重复调度能力。
- 题库和统计偏家长使用，练习、测试、背诵偏孩子使用。

## 首轮验收标准

只要满足下面条件，就算 MVP 成功：

1. 家长能录入一道语文、数学或英语题。
2. 家长能登录。
3. 家长能创建并切换多个孩子。
4. 系统能把题目加入指定孩子的今日练习。
5. 孩子能完成一次练习。
6. 系统能记录对错。
7. 答错的题能进入错题本。
8. 首页能看到今天练了多少题。
9. 数学口算能自动判分。
10. 语文和背诵题能手动判定。
11. 家长能设置每日目标题数和学习时长。
12. 孩子能完成一次作业后测试。
13. 完成目标后能获得小奖牌、贴纸、小星星或积分。

## 当前结论

RecallFlow 第一阶段不要追求“智能”，而要先做到“每天真的能用”。

最小产品应该围绕这条链路：

> 家长录入内容 → 孩子练习 → 系统记录结果 → 错题再次出现 → 家长看到掌握情况

这条链路跑通后，再扩学科、题型、调度算法和统计报表。
